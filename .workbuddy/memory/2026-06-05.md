## 根因发现 + Data URL 修复（09:50）

**根因**：Next.js 16 standalone 对所有请求返回 308 重定向死循环（非我们代码问题）。
- 验证：原始 server.js（无 monkey-patch）同样返回 308
- 影响：`/api/books/cover/[filename]` API 路由无法响应 → 封面图片永远加载失败

**解决方案**：绕过服务端，使用 base64 data URL 嵌入图片数据

**修改的文件**：
1. `src/app/api/books/parse-epub/route.ts` L180-183：封面 buffer → `data:${mimeType};base64,${buffer.toString('base64')}`
2. `src/lib/cover-url.ts` L14：新增 `if (url.startsWith('data:')) return url`
3. `src/components/reading-tracker/add-book-dialog.tsx` L830-835：data URL 输入框显示 `[已从 EPUB 提取封面]` 占位文字

**优缺点**：
- 优点：完全不依赖 Next.js 服务端，封面直接渲染
- 缺点：data URL 存储在 SQLite 中会使数据库变大（~270KB/本），对桌面应用可接受

**构建产物**：`src-tauri/target/release/bundle/macos/MyLibrary.app`（刚构建完成）

## 数据持久化修复（14:40）— 解决 App 重启后数据丢失

**根因**：`standalone.tar.gz` 中包含了构建时的 `data/library.db` 和 `public/uploads/*` 文件。每次 App 启动 tar 解压时，会用构建时的旧数据覆盖运行时产生的用户数据。

**修复方案**：将数据与代码分离，数据存储在独立的 `persistent/` 目录中。

### 修改的文件

1. **`src-tauri/src/lib.rs`**：新增数据持久化逻辑
   - 创建 `app_data_dir/persistent/data/` 和 `app_data_dir/persistent/uploads/` 目录
   - 启动时检测 `standalone/data/library.db` 并迁移到持久化目录
   - 迁移旧 `public/uploads/` 文件到持久化目录
   - 提取 tar 后复制种子 JSON 到持久化目录
   - 通过 `DATA_DIR` 和 `UPLOADS_DIR` 环境变量传递给 Node.js

2. **`src/lib/db.ts`**：`DATA_DIR = process.env.DATA_DIR || path.join(process.cwd(), 'data')`

3. **`src/lib/storage.ts`**：同上

4. **`src/lib/upload.ts`**：`uploadDir = process.env.UPLOADS_DIR || path.join(process.cwd(), 'public', 'uploads')`

5. **`src/app/api/books/cover/[filename]/route.ts`**：使用 UPLOADS_DIR 环境变量

6. **`src-tauri/tauri.conf.json`**：beforeBuildCommand 新增清理 `library.db`、`uploads/*`、`src-tauri/` 残留

### 目录结构（持久化后）
```
~/Library/Application Support/com.mylibrary.app/
├── standalone/          # 代码（每次启动从 tar.gz 解压更新）
│   ├── server.js
│   └── .next/
└── persistent/          # 数据（永久保留，不会被覆盖）
    ├── data/
    │   └── library.db
    └── uploads/
        └── *.jpg
```
